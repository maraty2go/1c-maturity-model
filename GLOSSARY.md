# Глоссарий терминов модели зрелости

В модели зрелости используются термины из различных областей: разработки 1С, DevOps, SRE, тестирования и информационной безопасности. Этот глоссарий поможет быстро разобраться в значениях.

---

## Основные термины

| Термин | Расшифровка | Краткое пояснение |
|--------|--------------|-------------------|
| **ADR** | Architecture Decision Record | Документирование архитектурных решений с описанием контекста, последствий и альтернатив. Хранятся в Git вместе с кодом. |
| **APDEX** | Application Performance Index | Метрика удовлетворённости пользователей производительностью приложения. Значение от 0 до 1, где 0.8–1.0 — отлично. |
| **CI/CD** | Continuous Integration / Continuous Delivery | Практики непрерывной интеграции (автоматическая сборка и тестирование) и непрерывной доставки (автоматическое развертывание). |
| **DAST** | Dynamic Application Security Testing | Динамический анализ безопасности — поиск уязвимостей в работающем приложении (например, через сканирование веб-интерфейсов). |
| **DRP** | Disaster Recovery Plan | План аварийного восстановления — документированная процедура действий при серьёзных сбоях (пожар, отказ ЦОД). |
| **EDA** | Event-Driven Architecture | Событийно-ориентированная архитектура, где компоненты обмениваются событиями через шину (RabbitMQ, Kafka). |
| **IaC** | Infrastructure as Code | Управление инфраструктурой через код (Ansible, Terraform, PowerShell). Позволяет версионировать настройки серверов. |
| **MLOps** | Machine Learning Operations | Практики управления жизненным циклом моделей машинного обучения: обучение, развертывание, мониторинг. |
| **MTTA** | Mean Time to Acknowledge | Среднее время от возникновения инцидента до начала реакции дежурного специалиста. |
| **MTTR** | Mean Time to Resolve | Среднее время полного восстановления после инцидента (от момента обнаружения до устранения). |
| **Postmortem** | — | Разбор инцидента с фокусом на извлечение уроков, без поиска виноватых (blameless culture). |
| **RPO** | Recovery Point Objective | Допустимый объём потери данных — максимальный период времени, за который данные могут быть утеряны при сбое. |
| **RTO** | Recovery Time Objective | Целевое время восстановления после сбоя — максимально допустимое время простоя системы. |
| **Runbook** | — | Документированная инструкция по действиям в нештатной ситуации (пошаговое руководство для дежурного). |
| **SAST** | Static Application Security Testing | Статический анализ исходного кода на уязвимости (без запуска программы). |
| **SCA** | Software Composition Analysis | Анализ используемых библиотек и компонентов на наличие известных уязвимостей. |
| **SLO** | Service Level Objective | Целевой уровень качества сервиса (например, 99.5% доступности, время ответа < 1 с). |
| **SLA** | Service Level Agreement | Соглашение с заказчиком об уровне обслуживания (юридически значимый документ). |
| **SRE** | Site Reliability Engineering | Инженерная дисциплина, объединяющая разработку и эксплуатацию для обеспечения надёжности сервисов. |

## Термины экосистемы 1С

| Термин | Пояснение |
|--------|-----------|
| **1С:АПК** | 1С:Автоматизированная проверка конфигураций — инструмент для статического анализа кода 1С, выявления ошибок и несоответствий стандартам. |
| **gitsync** | Инструмент (утилита или скрипт) для двусторонней синхронизации хранилища 1С с Git-репозиторием. |
| **opm** | OneScript Package Manager — пакетный менеджер для управления библиотеками на языке OneScript (аналог npm, pip). |
| **edtplugin** | Плагины для управления расширениями и зависимостями в среде 1С:EDT. |
| **Vanessa-Automation** | Open-source фреймворк для автоматизированного тестирования 1С (поведенческие тесты на базе Gherkin). |
| **xUnitFor1C** | Фреймворк для модульного тестирования 1С (аналог xUnit для других языков). |
| **Vanessa-Runner** | Инструмент для запуска тестов Vanessa-Automation в пакетном режиме (в CI/CD). |
| **OneScript** | Скриптовый язык, совместимый с синтаксисом 1С, для автоматизации задач вне платформы. |
| **Расширения** | Механизм 1С для точечных изменений конфигурации без снятия с поддержки. |
| **Функциональные переключатели** | Механизм включения/отключения новой логики через константы или регистры сведений без изменения конфигурации. |

## Процессы и методологии

| Термин | Пояснение |
|--------|-----------|
| **DevSecOps** | Подход, встраивающий практики безопасности на всех этапах разработки и эксплуатации. |
| **Shift-left** | Смещение проверок (тестирования, безопасности) на ранние этапы разработки. |
| **Self-healing** | Способность системы автоматически восстанавливаться после сбоя (перезапуск, переключение на резерв). |
| **Game Days** | Учения, в ходе которых искусственно создаются сбои для проверки устойчивости системы и готовности команды. |
| **Blameless Postmortem** | Культура разбора инцидентов без поиска виноватых, с фокусом на системные улучшения. |
| **DORA-метрики** | Набор метрик для оценки эффективности DevOps: частота релизов, время выполнения изменений, время восстановления, процент неудачных изменений. |
| **OKR** | Objectives and Key Results — методология постановки целей и ключевых результатов. |

## Инструменты, упоминаемые в модели

| Термин | Пояснение |
|--------|-----------|
| **SonarQube** | Платформа для статического анализа кода и управления качеством. |
| **Prometheus / VictoriaMetrics** | Системы сбора и хранения метрик (time series databases). |
| **Grafana** | Инструмент для визуализации метрик и создания дашбордов. |
| **Zabbix** | Система мониторинга инфраструктуры и сервисов. |
| **Perfmon** | Оснастка Windows Performance Monitor для сбора системных счетчиков. |
| **ELK / OpenSearch** | Стек для сбора, поиска и анализа логов (Elasticsearch, Logstash, Kibana / OpenSearch). |
| **Sentry** | Система сбора и анализа ошибок (исключений) в приложениях. |
| **RabbitMQ / Kafka** | Брокеры сообщений для асинхронных интеграций. |
| **1С:Шина** | Официальный продукт 1С для интеграции. |
| **HashiCorp Vault** | Инструмент для централизованного управления секретами (пароли, ключи, токены). |
| **Ansible / Terraform** | Инструменты для управления конфигурацией (Ansible) и инфраструктурой как кодом (Terraform). |
